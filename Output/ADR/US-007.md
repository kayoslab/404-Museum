## ADR: US-007 – Generate Believable Fake Names

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a name generator system under `src/domain/generators/` with curated fragment datasets in `src/domain/data/name-fragments.ts`, a `GeneratedName` type in `src/domain/types.ts`, brand-safety filtering, and 20+ snapshot tests.

### Architectural Assessment

**Alignment with project architecture:** The plan follows the prescribed `src/domain/` structure exactly. Generators go in `src/domain/generators/`, datasets in `src/domain/data/`, and types in `src/domain/types.ts`. This creates the `generators/` directory for the first time, establishing the pattern that future generator tickets (site content, layout selection, etc.) will follow.

**Dependency chain:** US-007 depends on US-004 (seeded random) and US-006 (datasets), both of which are complete and merged. The plan correctly consumes the `SeededRandom` interface, `createRng()`, `deriveSeed()`, `rng.pick()`, and `rng.bool()` from US-004, and the `Archetype` and `Region` types from US-006. All referenced APIs exist in the current codebase.

**Generator design:** The `generateName(rng, archetype, region)` signature is clean and composable. Using `deriveSeed()` for sub-generation steps maintains RNG isolation, which is critical for the project's determinism guarantee (Seed → identical output). The ~60% German fragment blending probability for `de-de` regions via `rng.bool(0.6)` is appropriately controlled by the seeded RNG, ensuring reproducibility.

**TypeScript strictness:** The plan must account for `noUncheckedIndexedAccess: true` when indexing into fragment arrays. Since `rng.pick()` already handles bounds-checking internally (it was designed for this in US-004), direct usage is safe. Any raw array indexing in fragment selection must guard against `undefined`. The `noUnusedLocals` and `noUnusedParameters` settings are correctly identified as a risk — the plan should ensure all helpers and parameters are consumed.

**Brand-safety filter:** The denylist approach with a 5-attempt re-roll cap and generic fallback is pragmatic. The cap prevents infinite loops while the denylist prevents obvious trademark issues. The risk of an incomplete denylist is real but acceptable — snapshot tests provide a manual review checkpoint, and the denylist can grow over time. Lowercased substring matching is the right granularity; exact-match would miss compounds like "EuroGoogle".

**Fragment dataset sizing:** The plan should ensure at least 10-15 fragments per category per archetype (as noted in risks). With 8 archetypes, each needing prefixes, roots, and suffixes, plus German variants, the `name-fragments.ts` file will be substantial but is purely static data with no runtime cost. The `as const` + `readonly` pattern matches existing datasets in `eras.ts`, `regions.ts`, and `archetypes.ts`.

**Barrel exports:** Creating `src/domain/generators/index.ts` and updating `src/domain/data/index.ts` follows the established pattern. The plan does not mention creating `src/domain/index.ts` (which doesn't exist yet) — this is fine; it can be deferred to a later ticket that wires generators into the rendering pipeline.

**Bundle impact:** Zero new dependencies. Static data arrays and pure functions. Negligible bundle cost.

**Key risks acknowledged:**
- Fragment list repetitiveness is mitigated by requiring 10-15 fragments per category — sufficient combinatorial variety for the current archetype count.
- German name blending could feel forced if it uses random German words rather than authentic business-naming patterns. The plan correctly specifies compound words and geographic roots (Luft, Stern, Netz, Werk, Haus) which reflect real German commercial naming conventions.
- Snapshot maintenance burden is controlled by capping at 20-25 focused cases rather than exhaustive coverage of the 120-combination matrix.
- The brand denylist is a living artifact — it should be easy to extend without changing generator logic, and the plan's placement within `name-fragments.ts` co-locates it with the data it guards.

**No architectural concerns identified.** This is a pure domain-layer addition with no side effects, no new dependencies, no build configuration changes, and no modifications to existing code beyond additive type extensions and barrel export updates. The generator establishes a clean, testable pattern for future content generation modules.
