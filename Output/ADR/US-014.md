## ADR: US-014 – Create Floating Refresh Button

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a floating refresh button in `src/ui/refresh-button.ts`, styled via `src/styles/refresh-button.css`, with a render extraction in `src/main.ts` to support re-generation without browser reload, and file-system-based tests following the established US-013/US-015 pattern.

### Architectural Assessment

**Alignment with project architecture:** The plan places the button component in `src/ui/` and styles in `src/styles/`, exactly matching the structure prescribed in CLAUDE.md and the patterns established by US-013 (info button) and US-015 (share button). The `createRefreshButton(container): HTMLButtonElement` factory function signature is identical to `createInfoButton` and `createShareButton`, maintaining API consistency across all floating controls.

**Dependency chain:** US-014 depends on US-003 (shell structure), US-005 (URL seed), and US-011 (content modules), all complete and merged. The plan correctly consumes:
- `#overlay-ui` container from US-003 with its `pointer-events: none` + `z-index: 10` pattern — the refresh button must set `pointer-events: auto` as established by info and share buttons
- `generateSeed()` from `src/domain/seed.ts` — confirmed exported, generates 12-char random seeds via `Math.random()`
- `writeSeedToUrl(seed)` from `src/domain/url-seed.ts` — confirmed, uses `history.replaceState` to update the URL without navigation
- `generateSite(seed)` from `src/domain/generate-site.ts` — confirmed, returns `GeneratedSite` from a seed string
- `renderHomepage(container, site)` from `src/render/render-homepage.ts` — confirmed, clears container and renders the full page structure

**Render extraction design:** The plan's proposal to extract a `renderSite(container)` function from `src/main.ts` is the key architectural decision. The current `main.ts` has seed resolution (lines 43-45) and rendering (lines 47-50) as sequential inline operations. Extracting a reusable render path that calls `generateSeed()` → `writeSeedToUrl(seed)` → `generateSite(seed)` → `renderHomepage(container, site)` is clean and minimal. Critically, the plan correctly preserves the initial load path (`resolveSeed(readSeedFromUrl())`) for seed-from-URL behavior, only using `generateSeed()` for refresh clicks. This distinction is essential — without it, refreshing the page would lose the ability to share seeded URLs.

**No duplicate controls guarantee (AC #5):** The current `main.ts` structure naturally prevents duplicate controls. Overlay UI elements (info button, share button, modal, toast) are created once in the `if (!generatedSiteContainer || !overlayUi) { } else { ... }` block (lines 24-40). The render path only touches `generatedSiteContainer` (lines 47-50). Since `renderHomepage` clears the container via the `generated-page` wrapper replacement pattern, repeated calls regenerate content without affecting the overlay layer. The refresh button will be created once alongside the other controls, and its click handler will only invoke the extracted render function on `generatedSiteContainer`. No architectural risk of duplicate controls.

**Button stacking position:** The plan specifies `bottom: 136px; right: 24px;` which follows the established 56px vertical spacing pattern:
- Info button: `bottom: 24px` (confirmed in `info-button.css`)
- Share button: `bottom: 80px` (confirmed in `share-button.css`)
- Refresh button: `bottom: 136px` (56px above share)

This is consistent and leaves adequate spacing. The toast component at `bottom: 160px` (confirmed in `toast.css`) sits 24px above the refresh button — tight but non-overlapping since the toast is horizontally centered (`left: 50%; transform: translateX(-50%)`) while buttons are right-aligned (`right: 24px`). If the toast needs adjustment for visual clearance, it's a CSS-only change with no architectural impact.

**RNG and determinism:** The refresh click handler generates a fresh seed via `generateSeed()`, which uses `Math.random()` — this is the one sanctioned use of `Math.random()` in the codebase (established in US-004). The new seed flows through `generateSite(seed)` which uses `deriveSeed` for all internal RNG streams, maintaining the determinism guarantee: any generated seed will always produce the same site if re-entered via URL.

**URL update via replaceState:** Using `writeSeedToUrl(seed)` on refresh correctly updates the URL without creating browser history entries. This means the browser back button won't cycle through generated seeds — the same behavior established on initial load. This is the right choice: seed generation is not a navigation event.

**Rapid click risk:** The plan identifies rapid clicking as a risk and proposes disabling the button during render or debouncing. Since `renderHomepage` is synchronous DOM manipulation (no async operations), the risk of race conditions is minimal — each click will complete its render before the next event loop tick processes the next click. However, rapid clicks could cause visible flickering. A simple guard (e.g., `if (rendering) return`) is a proportionate mitigation. This is an implementation detail, not an architectural concern.

**Test strategy:** The plan's test file follows the established US-013/US-015 pattern:
- File existence checks for `refresh-button.ts` and `refresh-button.css`
- CSS validation: `position: absolute`, `bottom: 136px`, `right: 24px`, `pointer-events: auto`, min 44px dimensions
- Button validation: `aria-label`, class name, text content
- Main.ts integration: import checks for CSS and component
- Build verification: `tsc --noEmit` and `npm run build`

This is consistent with the established testing philosophy where structural correctness is validated via file-system tests and runtime behavior is deferred to manual testing and future E2E tests.

**TypeScript strictness:** No indexed access is involved in this plan — the refresh button is pure DOM creation and event wiring. The `generateSeed()`, `writeSeedToUrl()`, `generateSite()`, and `renderHomepage()` functions all have well-typed signatures with no `undefined` concerns. No `noUncheckedIndexedAccess` issues.

**Bundle impact:** Zero new dependencies. One TypeScript module and one CSS file. Negligible bundle cost.

**Key risks acknowledged:**
- Rapid clicking causing visual flicker is the primary UX risk. The synchronous nature of DOM rendering makes race conditions unlikely, but a simple rendering guard is proportionate mitigation.
- Button stacking with three controls (info, share, refresh) is manageable at the current count. The plan correctly notes that a layout system may be needed if more buttons are added — this is a future concern, not a blocker.
- The extracted render function must only operate on `#generated-site`, never on `#overlay-ui` — the current code structure enforces this by design, and the plan's description is consistent with this separation.
- Toast vertical position (160px) is close to the refresh button (136px + 44px = 180px top edge) but non-overlapping due to different horizontal alignment. No conflict.

**No architectural concerns identified.** This is a straightforward UI feature that extends the floating controls system established by US-013 and US-015. The render extraction in main.ts is the only structural change, and it is minimal — converting inline sequential calls into a reusable function. The plan follows all established patterns for factory functions, CSS styling, overlay integration, and testing. No new dependencies, no build configuration changes, and no modifications to the domain or render layers.
