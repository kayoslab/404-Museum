## ADR: US-012 – Add Abandoned and Broken Historical Details

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement an abandonment detail system with a new `AbandonmentDetail` type in `src/domain/types.ts`, curated era-keyed fragment datasets in `src/domain/data/abandonment-fragments.ts`, a deterministic generator in `src/domain/generators/abandonment-generator.ts`, a rendering function in `src/render/render-abandonment-details.ts`, CSS styling in `src/styles/generated-site.css`, and comprehensive tests covering determinism, count bounds, era compatibility, and content readability.

### Architectural Assessment

**Alignment with project architecture:** The plan places all new code exactly where CLAUDE.md prescribes — types in `src/domain/types.ts`, datasets in `src/domain/data/`, the generator in `src/domain/generators/`, the renderer in `src/render/`, and styles in `src/styles/`. This follows every pattern established by US-007 through US-011 without introducing new structural conventions.

**Dependency chain:** US-012 depends on US-008 (metadata generator) and US-011 (content modules), both complete and merged. The plan correctly consumes:
- `deriveSeed(parentSeed, key)` and `createRng(seed)` from `src/domain/seed.ts` — confirmed signatures match
- `GeneratedSite` interface from `src/domain/generate-site.ts` — confirmed with all 6 existing fields
- `renderHomepage` pipeline in `src/render/render-homepage.ts` — confirmed flow: header → hero → modules → footer
- `renderFooter(site)` from `src/render/render-footer.ts` — confirmed, currently displays `site.metadata.yearFounded` as copyright year
- `metadata.yearFounded` from `SiteMetadata` — confirmed, used for copyright year derivation in abandonment context

**GeneratedSite interface extension:** Adding `abandonmentDetails: readonly AbandonmentDetail[]` to the `GeneratedSite` interface is a breaking change to the type contract. All consumers that destructure or serialize `GeneratedSite` will see a new field. The impact is well-contained:
- `renderHomepage` receives `GeneratedSite` but only passes it through to sub-renderers — no structural change needed
- `renderHeader`, `renderHero`, `renderFooter` all take `GeneratedSite` and access only the fields they need — TypeScript's structural typing means they continue to compile without modification unless they explicitly check for the new field
- Content module `render(site, rng)` functions similarly access only their needed fields
- Existing snapshot tests for US-008 and US-010/011 that serialize `GeneratedSite` will break and require snapshot updates — the plan acknowledges this risk

The key architectural decision is placing `abandonmentDetails` on `GeneratedSite` rather than computing them at render time. This is the correct choice: it maintains the project's established pattern where all generative decisions happen in the domain layer (`generate-site.ts`) and the render layer is a pure function of domain data. It also preserves the determinism guarantee — abandonment details are derived from the seed and become part of the reproducible site specification.

**RNG isolation:** The plan uses `deriveSeed(seed, 'abandonment')` for the generator's RNG stream. This follows the established convention:
- Era selection: `deriveSeed(seed, 'era')`
- Archetype selection: `deriveSeed(seed, 'archetype')`
- Region selection: `deriveSeed(seed, 'region')`
- Theme: `deriveSeed(seed, 'theme')` inside `selectTheme`
- Metadata: `deriveSeed(seed, 'metadata')` inside `generateMetadata`
- Module selection: `deriveSeed(seed, 'modules')` in `selectModules`
- Per-module rendering: `deriveSeed(site.seed, mod.id)` in `renderHomepage`

Adding `'abandonment'` as a new key creates an independent stream that cannot perturb any existing generation. This is the critical correctness property — without it, adding the abandonment generator would change the output of every existing generator for the same seed, breaking all snapshot tests.

**Renderer coupling analysis:** The plan proposes two rendering strategies for abandonment details:

1. **Standalone elements** (broken badges, dead counters, browser notices, missing asset placeholders, expired SSL notices, defunct partner logos): These are new DOM elements injected by `render-abandonment-details.ts` into the page. This is architecturally clean — no coupling to existing renderers.

2. **Modifications to existing sections** (outdated copyright year → footer, stale news dates → news-block module): These create coupling between the abandonment system and specific renderers. Since `abandonmentDetails` lives on `GeneratedSite`, `renderFooter` can read `site.abandonmentDetails` to conditionally override the copyright year without any signature change — it already receives the full `site` object. Similarly, the news-block module receives `site` via its `render(site, rng)` call. This is read-only coupling through the established data contract, which is acceptable.

The alternative — having `render-abandonment-details.ts` post-hoc manipulate DOM elements created by other renderers (e.g., querying `.footer-copyright` and rewriting its text) — would be brittle and dependent on internal DOM structure. The plan's approach of passing abandonment context through `GeneratedSite` and letting individual renderers consume it is architecturally superior. The plan should ensure the footer and news-block modifications are implemented as opt-in checks within those renderers (reading from `site.abandonmentDetails`) rather than external DOM manipulation.

**Generator design — seed-based signature:** The plan follows the `generateMetadata` pattern by accepting `seed` (plus era, archetype, region) rather than `rng: SeededRandom`. This is correct for the same reason it was correct for metadata: the generator owns its RNG lifecycle via `deriveSeed(seed, 'abandonment')`, ensuring complete isolation. The additional parameters (era, archetype, region) are needed for era-compatible fragment filtering — the generator must know which era to filter fragments against.

**Template placeholder substitution:** The plan uses `{count}`, `{staleDate}`, `{staleYear}` placeholders in fragment templates, filled with deterministic values derived from `yearFounded`. This is a new pattern not used by existing fragment datasets (`name-fragments.ts`, `metadata-fragments.ts`, `module-fragments.ts` all use plain strings). The risk of template syntax leaking into rendered output (`{count}` appearing literally) is real but straightforward to mitigate — a simple `replace()` pass with a fallback empty string for unrecognized placeholders prevents leakage. The plan should ensure all placeholder keys are exhaustively handled and tested.

**Era fragment count sufficiency:** The 2–5 count requirement means each era must have at least 5 compatible fragments (to support the maximum). With 8 detail types (outdated-copyright, stale-news-date, broken-badge, dead-counter, deprecated-browser-notice, missing-asset-placeholder, expired-ssl-notice, defunct-partner-logo), not all are appropriate for all eras — e.g., expired-ssl-notice is anachronistic for early-web sites. The plan must ensure at least 5 era-compatible fragments exist per era. Given 8 detail types and 5 eras, the dataset needs careful curation to avoid under-representation for any era. The plan's risk section identifies this correctly.

**CSS architecture:** Adding `.abandonment-*` prefixed classes to `generated-site.css` follows the established module-prefix convention (`.module-features-grid`, `.module-news-block`, etc.). Using CSS custom properties for theme consistency (colors, borders, spacing) ensures abandonment elements inherit the era-appropriate visual style. The desaturated/faded treatment for broken badges and defunct logos is a CSS-only effect (`filter: grayscale()`, `opacity`) that requires no JavaScript — consistent with the project's preference for CSS-generated visuals over image assets.

**Mobile responsiveness:** The plan must ensure abandonment detail elements (counter widgets, badge strips, browser notice banners) are responsive. The existing `generated-site.css` includes a `@media (max-width: 600px)` breakpoint for module-specific adjustments. Abandonment elements should follow this pattern. The plan's risk section identifies this correctly.

**Content readability (AC #3):** The acceptance criterion that "main content remains readable and usable" requires that abandonment elements are decorative overlays or peripheral elements, not content blockers. The plan's rendering strategy places most details in expected locations — counter widgets in sidebars or footers, browser notices as top banners, broken badges as decorative strips. The test plan includes a DOM test verifying `.site-body` text content is not obscured. This is sufficient coverage for the readability criterion.

**No real network failures (AC #4):** All broken effects are CSS/DOM-only simulations. Broken badges use greyed-out placeholder divs, not actual broken `<img>` tags with invalid `src` attributes. Missing asset placeholders show a dashed-border box with "[Image not found]" text. This is the correct approach — no `fetch()` or `XMLHttpRequest` calls are made, and no browser console errors are produced from simulated failures. The test plan includes verification that no actual network calls are made.

**TypeScript strictness:** `noUncheckedIndexedAccess: true` applies to era-keyed fragment lookups in `abandonment-fragments.ts`. If fragments are stored as `Record<string, readonly string[]>` keyed by era ID, looking up `fragments[era.id]` returns `readonly string[] | undefined`. The plan must handle this — either via a fallback array (matching the `summaryTemplates` + `fallbackSummaryTemplates` pattern from `metadata-fragments.ts`) or by using a validated lookup. Since `rng.pick()` already handles the `T | undefined` → `T` cast internally, the primary concern is the record lookup itself.

**Test strategy:** The plan's test file covers the essential verification points:
- File structure verification — matches US-010/011 pattern
- Generator count bounds (2–5) across all era/archetype combos — directly tests AC #1
- Determinism (50 iterations) — matches established rigor from US-004/007/008
- Era compatibility — ensures early-web sites don't get startup-era details
- No DOM dependencies in generator — confirms domain/render separation
- Snapshot stability — regression coverage
- Content readability DOM test — directly tests AC #3

The test plan is comprehensive and follows established patterns.

**Bundle impact:** Zero new dependencies. One dataset file, one generator function, one renderer function, and CSS additions. All pure functions and static data. Negligible bundle cost.

**Key risks acknowledged:**
- `GeneratedSite` interface change breaks existing snapshot tests for US-008 and US-010/011. This is expected and manageable — snapshot updates are routine. The plan acknowledges this.
- Template placeholder substitution must handle all edge cases to prevent `{count}` or `{staleDate}` appearing in rendered output. A comprehensive test covering all template patterns mitigates this.
- Era fragment sufficiency (at least 5 per era) must be validated during dataset curation. Insufficient fragments would cause the generator to produce fewer than the maximum 5 details for some eras, but the 2–5 range accommodates this gracefully.
- Footer/news-block coupling through `GeneratedSite` is read-only and follows established patterns. Changes to footer structure would require coordinated updates to abandonment rendering, but this is the same coupling pattern that already exists between footer and `SiteMetadata`.
- Mobile responsiveness for abandonment CSS elements requires explicit responsive rules, consistent with existing module styling.

**No architectural concerns identified.** This plan extends the established domain → render pipeline with a new generation step and rendering layer, using proven patterns for RNG isolation, type extension, dataset curation, and CSS theming. The coupling between abandonment details and existing renderers (footer, news-block) is mediated through the `GeneratedSite` data contract, which is the architecturally correct approach. No new dependencies, no build configuration changes, and no structural changes to existing code beyond the additive `abandonmentDetails` field on `GeneratedSite`.
