## ADR: US-011 – Render Reusable Homepage Content Modules

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a content module system consisting of a module registry in `src/domain/data/content-modules.ts`, a seeded module selector in `src/domain/generators/module-selector.ts`, fragment datasets in `src/domain/data/module-fragments.ts`, six module renderers in `src/render/modules/`, a CSS file in `src/styles/content-modules.css`, and integration into the existing `renderHomepage` orchestrator.

### Architectural Assessment

**Alignment with project architecture:** The plan places code exactly where CLAUDE.md prescribes — data definitions in `src/domain/data/`, generator logic in `src/domain/generators/`, renderers in `src/render/`, and styles in `src/styles/`. The new `src/render/modules/` subdirectory is a well-motivated organizational choice given the six new renderer files; it mirrors the flat-file pattern in `src/render/` but adds a level of grouping to prevent the directory from becoming crowded as the project grows.

**Dependency chain:** US-011 depends on US-010, which is complete and merged. The plan correctly consumes:
- `deriveSeed(parentSeed, key)` and `createRng(seed)` from `src/domain/seed.ts` — confirmed signatures match
- `GeneratedSite` interface from `src/domain/generate-site.ts` — confirmed: `{ seed, era, archetype, region, theme, metadata }`
- `renderHomepage` from `src/render/render-homepage.ts` — confirmed integration point: modules insert between `renderHero()` and `renderFooter()` calls
- Archetype/Era types from `src/domain/types.ts` — confirmed with `id: string` fields for compatibility filtering

**ContentModule type design:** The `ContentModule` interface with `compatibleArchetypeIds` and `compatibleEraIds` fields introduces a new filtering pattern — modules declaring which archetypes/eras they support, rather than the existing pattern where archetypes declare compatible eras. This inverted relationship is appropriate here: modules are the new concept being registered, and they should declare their own constraints rather than modifying the existing Archetype/Era types. The `'*'` wildcard for universal compatibility is a pragmatic convention that avoids listing all 8 archetype IDs and 5 era IDs on every universal module.

**Minimum-2 guarantee analysis:** This is the most important correctness property. The plan defines 6 modules: `features-grid` (universal), `news-block` (universal, implied), `testimonials` (universal, implied), `download-panel` (os-vendor, media-startup), `pricing-table` (e-commerce, os-vendor, webmail), `guild-roster` (mmorpg-guild only). With at least 3 universal modules, every archetype+era combination will have a minimum of 3 compatible modules available — the minimum-2 selection guarantee holds. The risk note about guild-roster being archetype-specific is correctly identified, but the universal module count makes this a non-issue. The plan should ensure the selector explicitly asserts or guards against fewer than 2 compatible modules as a defensive measure, even though current data prevents this.

**Module selector determinism:** Using `deriveSeed(seed, 'modules')` to create an isolated RNG for module selection follows the established pattern from `generateSite`, `renderHeader`, and `generateMetadata`. This ensures module selection is independent of other generation steps — adding or removing modules cannot affect archetype/era/region selection or metadata generation. The filter-shuffle-slice algorithm (filter compatible → shuffle with RNG → pick 2-3) is the same pattern used in `renderHeader` for nav item selection (lines 51-54 of render-header.ts), which is proven and correct.

**Renderer pattern consistency:** Each module renderer follows the exact pattern established by `renderHeader`, `renderHero`, and `renderFooter` — a function taking `GeneratedSite` and returning `HTMLElement`. The `deriveSeed(site.seed, 'module-name')` pattern for per-module RNG isolation matches the existing `deriveSeed(site.seed, 'nav')` usage in `renderHeader`. This is architecturally consistent and predictable for future contributors.

**Module dispatcher design:** The `Record<string, (site: GeneratedSite) => HTMLElement>` mapping in `src/render/modules/index.ts` is a clean registry pattern. It decouples the selection step (which module IDs?) from the rendering step (how to render each ID?), allowing new modules to be added by: (1) adding a descriptor to the registry, (2) adding a renderer function, and (3) adding an entry to the dispatcher map. This extensibility pattern is appropriate given the ticket's note that "at least 2 reusable body content modules exist" — the architecture should support easy addition of more.

**Integration with renderHomepage:** The current `renderHomepage` (confirmed in `src/render/render-homepage.ts`) appends three elements in order: `renderHeader(site)`, `renderHero(site)`, `renderFooter(site)`. The plan inserts modules between hero and footer by calling `selectModules` and `renderModules` and appending each returned element before the footer. This is a minimal, clean modification — the hero's `flex: 1` CSS (which makes the hero fill available vertical space) will need adjustment since modules now occupy space between hero and footer. The plan's CSS step should address this by moving `flex: 1` to a wrapper or distributing space differently. This is a CSS concern, not an architectural one, but worth flagging for implementation.

**Hero flex: 1 interaction:** The current `.site-hero { flex: 1 }` in `generated-site.css` makes the hero expand to fill all remaining vertical space between header and footer. With modules inserted after the hero, this property means the hero will still try to claim all available space, pushing modules to the bottom. The implementation must either remove `flex: 1` from the hero, move it to a content wrapper, or adjust the layout strategy. This is not called out in the plan but is a straightforward CSS fix that does not affect the architectural design.

**CSS custom property compliance:** The plan correctly specifies that all module styles must use existing CSS custom properties (`--color-primary`, `--font-family`, `--spacing-*`, `--border-*`, `--color-surface`, etc.). These properties are set by `applyTheme()` in `apply-theme.ts` and are confirmed to cover typography, palette, spacing, borders, and surfaces. All existing component styles (`site-header`, `site-hero`, `site-footer`) use these same properties with fallback values — the plan's modules should follow the identical pattern.

**Responsive breakpoint consistency:** The plan specifies responsive styles at the existing 600px breakpoint, matching the `@media (max-width: 600px)` rule already in `generated-site.css`. This is correct.

**US-010 snapshot impact:** The plan correctly identifies that existing US-010 snapshot tests will break because `renderHomepage` now produces additional DOM nodes (modules between hero and footer). The US-010 test file (`tests/us-010-render-homepage.test.ts`) includes snapshot tests for specific seeds. These will need updating — this is expected and acceptable. The structural tests (checking for `header`, `section.site-hero`, `footer.site-footer` presence) should remain passing since modules are additive.

**Fragment dataset sizing risk:** The plan's risk about needing 8-10 entries per archetype per module type is the primary content-quality risk. With 6 modules × 8 archetypes, the `module-fragments.ts` file could be substantial (potentially 500+ lines of curated text data). This is consistent with the existing `name-fragments.ts` and `metadata-fragments.ts` pattern but at larger scale. The data is purely static with no runtime cost, so this is a file-organization concern, not a performance concern.

**TypeScript strictness:** With `noUncheckedIndexedAccess: true`, the module dispatcher's `Record<string, ...>` lookup returns `Function | undefined`. The dispatcher must handle the case where a module ID has no corresponding renderer — either via a guard check or by using a `Map` with guaranteed entries. Since the module IDs come from the registry and the dispatcher map is populated from the same set of IDs, this should be safe in practice, but the type system will require explicit handling. The plan should use a non-null assertion with a comment or a runtime error for unknown module IDs.

**Bundle impact:** Zero new dependencies. Static data arrays and pure DOM functions. The CSS addition is the largest single file but follows existing patterns. Negligible bundle cost.

**Key risks acknowledged:**
- Fragment dataset variety (8-10 entries per archetype per module) is correctly identified as a content-quality risk. The combinatorial explosion is manageable given that many modules are universal and can share fragments across archetypes.
- CSS consistency across 6 module types is the largest styling addition so far. Mitigated by strict adherence to existing CSS custom properties and the proven `generated-site.css` pattern.
- Guild-roster archetype specificity cannot violate the minimum-2 guarantee because at least 3 universal modules exist. A defensive assertion in the selector would add safety.
- US-010 snapshot updates are expected and correctly called out. No structural tests should break.

**Minor implementation note:** The hero's `flex: 1` CSS property will interact with the new modules and likely needs adjustment when modules are inserted. This is not an architectural concern but should be addressed during CSS implementation to avoid layout issues.

**No architectural concerns identified.** This plan extends the existing render layer with a well-motivated module pattern that mirrors established conventions (seeded RNG isolation, factory functions returning HTMLElement, CSS custom property theming, compatibility filtering). The new `src/render/modules/` subdirectory is a clean organizational boundary. The module registry + selector + dispatcher architecture provides a clear extension path for adding future modules without modifying existing code. No new dependencies, no build configuration changes, and the only modification to existing code is the expected integration point in `renderHomepage`.
