## ADR: US-016 – Add Responsive Mobile Support

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to create a centralized `src/styles/responsive.css` file containing all mobile breakpoint overrides for floating controls, typography, info modal, toast, roster table, and horizontal overflow prevention, imported last in `src/main.ts`, with the existing 600px media query consolidated from `generated-site.css`.

### Architectural Assessment

**Alignment with project architecture:** The plan places all new CSS in `src/styles/`, following the established convention. Creating a dedicated `responsive.css` file rather than scattering media queries across individual CSS files is a sound organizational choice — it centralizes all breakpoint logic in one file, making mobile behavior discoverable and maintainable. The file is imported in `src/main.ts` alongside existing style imports, following the established pattern.

**Dependency chain:** US-016 depends on US-013 (info button), US-014 (refresh button), and US-015 (share button), all complete and merged. The plan correctly targets all three button CSS files (`info-button.css`, `share-button.css`, `refresh-button.css`), the toast (`toast.css`), and the info modal (`info-modal.css`) for responsive overrides. All referenced CSS classes and positioning values have been verified against the current codebase:
- Info button: `bottom: 24px; right: 24px` — confirmed in `info-button.css`
- Share button: `bottom: 80px; right: 24px` — confirmed in `share-button.css`
- Refresh button: `bottom: 136px; right: 24px` — confirmed in `refresh-button.css`
- Toast: `bottom: 160px; left: 50%; white-space: nowrap` — confirmed in `toast.css`
- Info modal: `max-width: 420px; width: 90%; padding: 32px` — confirmed in `info-modal.css`

**CSS cascade and specificity — key risk correctly identified:** Moving the existing `@media (max-width: 600px)` block from `generated-site.css` (lines 331-360) into `responsive.css` changes the cascade order. Currently, the media query lives within `generated-site.css` which is the last CSS import in `main.ts`. The plan specifies importing `responsive.css` last — this is critical and correct. Since all selectors in the media query block use the same specificity as the base rules (e.g., `.site-header`, `.site-hero`, `.features-grid`), the override behavior depends entirely on source order. Importing `responsive.css` after `generated-site.css` preserves the current behavior. The plan explicitly acknowledges this risk and specifies last-import ordering.

**Breakpoint strategy:** The two breakpoints — 600px (existing, consolidated) and 480px (new, small phones) — are well-chosen. The 600px breakpoint already handles tablet/large-phone layouts for content grids and padding. The 480px breakpoint targets small phones (320px–480px) where floating controls, toast, and modal need tighter spacing. This two-tier approach avoids over-engineering while covering the acceptance criteria range (320px upward).

**Floating control spacing adjustments:** Reducing `right` and `bottom` offsets from 24px to 16px at ≤480px reclaims 8px of edge space on each side — meaningful on a 320px viewport where 24px×2 = 48px (15% of width) is consumed by margins alone. The tighter vertical stacking (16px / 64px / 112px instead of 24px / 80px / 136px) reduces total button stack height from 180px to 156px, which is important on short viewport heights (e.g., iPhone SE at 568px in portrait). The 48px spacing between button bottoms maintains adequate separation for 44px touch targets.

**Toast and button overlap risk:** The plan identifies toast-button overlap as a risk. Currently the toast is at `bottom: 160px` centered horizontally, while buttons are right-aligned. At 480px width, the toast with `max-width: calc(100vw - 32px)` could extend to the right edge, overlapping with the button column. However, the toast is transient (auto-hides after a few seconds) and the buttons are below it vertically (highest button at 112px on mobile), so visual overlap is unlikely. The `white-space: normal` override for the toast at ≤480px is correct — the current `white-space: nowrap` would cause horizontal overflow on narrow screens with longer messages like "Link copied to clipboard!".

**Info modal mobile overrides:** Setting `width: 95%` at ≤480px (up from the base `width: 90%`) maximizes content area on small screens. The `max-height: 90vh; overflow-y: auto` addition prevents the modal from being taller than the viewport — this is a genuine gap in the current CSS, where the modal has no height constraint and could extend beyond the viewport on short screens. The padding reduction from 32px to 20px is proportionate.

**Roster table horizontal scroll:** The `overflow-x: auto` approach for the guild roster module's `.roster-table` parent is the correct pattern for wide tables on narrow viewports. This contains horizontal scrolling within the table wrapper rather than allowing it to cause page-level overflow. This is preferable to the alternative of forcing column hiding or responsive table reformatting, which would compromise the retro aesthetic.

**Defensive overflow-x: hidden:** The `max-width: 100vw; overflow-x: hidden` rule on `#generated-site` is a defensive measure against any content that escapes its container. This is appropriate given the variety of generated content (tables, long text strings, grid layouts) that could cause horizontal overflow on 320px viewports. The risk of clipping intentionally wide content (noted in the plan) is low — the era themes use CSS-generated visuals rather than fixed-width elements, and the existing grid layouts already use `auto-fit` with `minmax` which adapts to container width. Early-web table layouts are the most likely source of overflow, but these are styled with percentage widths in the current CSS.

**Typography minimum sizes:** Setting `font-size: 14px` minimum on `#generated-site` at ≤480px ensures readability without pinch zoom. The hero tagline scaling from 1.5em (at ≤600px, existing) to 1.25em (at ≤480px, new) is a proportionate reduction. The `.site-body h2` scaling to 1.25em matches. These values satisfy the acceptance criterion that "typography remains readable without pinch zoom" while preserving the era-appropriate styling at larger viewports.

**Test strategy:** The test file follows the established file-system-based pattern (US-013/014/015):
- File existence and import verification
- Media query presence via regex
- CSS rule assertions for specific responsive overrides
- Build verification via `tsc --noEmit` and `npm run build`

This is consistent with the project's testing philosophy. The manual testing recommendations (320px, 375px, 414px viewport testing in DevTools) appropriately cover the visual verification that automated tests cannot provide.

**Existing test compatibility:** The plan correctly notes that existing tests (US-013/014/015) assert specific pixel values for button positioning (e.g., `bottom: 24px`, `right: 24px`) by reading the base CSS files. Since responsive overrides live in a separate `responsive.css` file and only apply within `@media` blocks, the base CSS assertions remain valid. No existing tests need modification.

**Bundle impact:** Zero new dependencies. One CSS file with media query overrides. Negligible bundle cost.

**Key risks acknowledged:**
- CSS cascade order when moving the 600px media query is the highest-impact risk. The plan's explicit requirement to import `responsive.css` last in `main.ts` is the correct and sufficient mitigation. Implementation must verify this ordering.
- Toast-button overlap at narrow widths is a visual concern, not a functional one. The transient nature of the toast and the vertical separation make actual overlap unlikely.
- `overflow-x: hidden` on `#generated-site` could theoretically clip intentional wide content, but current era themes use percentage-based or auto-fit layouts that adapt to container width. The risk is minimal and the defensive benefit outweighs it.
- Early-web era tables are the most likely source of horizontal overflow at 320px. The roster table gets explicit `overflow-x: auto` treatment; other table content inherits the defensive `overflow-x: hidden` on the container, which is acceptable since these tables are decorative rather than data-critical.

**No architectural concerns identified.** This is a CSS-only change (plus one import line in `main.ts`) that introduces no new runtime logic, no new dependencies, and no modifications to existing TypeScript code. The centralized responsive file is a clean organizational pattern that keeps mobile overrides discoverable and maintainable. The plan correctly addresses all five acceptance criteria: 320px+ usability (breakpoints + overflow prevention), no horizontal overflow (defensive rule + toast fix), tappable non-overlapping controls (adjusted spacing), readable typography (minimum sizes), and functional retro layouts (content adapts while preserving era aesthetics).
