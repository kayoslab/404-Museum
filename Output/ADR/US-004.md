## ADR: US-004 – Deterministic Seeded Random Utility

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a deterministic seeded PRNG in `src/domain/seed.ts` with types in `src/domain/types.ts`, comprehensive unit tests, and a Math.random usage guard.

### Architectural Assessment

**Alignment with project architecture:** The plan places code exactly where CLAUDE.md prescribes — seed utilities under `src/domain/`, types in `src/domain/types.ts`, tests in `tests/`. This will be the first code in `src/domain/`, establishing the directory structure that US-006 (datasets) and future generator tickets will build on.

**Algorithm choice:** Mulberry32 is appropriate. 32-bit state is more than sufficient for generating page layouts, color choices, and text selections. The project is not doing cryptography or generating sequences long enough to expose period limitations. If a future ticket needs longer sequences, the `SeededRandom` interface abstracts the algorithm, making a swap straightforward.

**TypeScript strictness:** The plan correctly identifies that `noUncheckedIndexedAccess: true` in tsconfig.json means `pick()` must bounds-check before array access. This is the right approach — no suppression of the strict setting.

**Dependency on US-002:** US-002 is complete (merged). The tooling it established (Vitest, ESLint, TypeScript) is in place and the plan's final verification step (`npm run lint && npm run typecheck && npm run test`) uses exactly those scripts.

**Math.random guard:** The grep-based test is a pragmatic enforcement mechanism. A dedicated ESLint rule would be heavier than warranted at this stage. The distinction between `generateSeed()` (allowed to use Math.random for initial entropy) and generator code (must not) is clear and well-documented in the plan.

**Bundle impact:** Zero new dependencies. A single utility module with pure functions. Negligible bundle cost.

**Key risks acknowledged:**
- Hash collisions (two seed strings mapping to the same 32-bit integer) are theoretically possible but practically negligible for this use case. FNV-1a's distribution properties minimize the risk further.
- The `generateSeed()` / Math.random exception is clearly scoped and documented. The test guard will catch any accidental Math.random usage in generator modules.
- `deriveSeed()` design is sound — combining parent seed + key via hashing for independent sub-streams is the standard approach for deterministic generation with multiple modules.

**No architectural concerns identified.** This is a foundational utility with no side effects, no new dependencies, and no build configuration changes. The `SeededRandom` interface establishes a clean contract that all downstream generator modules will consume.
