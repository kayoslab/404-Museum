## ADR: US-015 – Create Floating Share Button

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a floating share button with toast feedback in `src/ui/`, styled via `src/styles/`, with share logic in `src/ui/share-action.ts`, wired through `src/main.ts`, and tested following the established file-system-based pattern from US-013.

### Architectural Assessment

**Alignment with project architecture:** The plan places all new files exactly where CLAUDE.md prescribes — UI components in `src/ui/`, styles in `src/styles/`. The four new files (`share-button.ts`, `share-action.ts`, `toast.ts`, `share-button.css`, `toast.css`) follow the same factory-function pattern established by US-013's `createInfoButton` and `createInfoModal`. This is the expected extension point for the floating controls system.

**Dependency chain:** US-015 depends on US-003 (shell structure) and US-005 (URL seed), both complete and merged. The plan consumes the `#overlay-ui` container from US-003 and relies on `window.location.href` containing the seed parameter established by US-005. The current `src/main.ts` confirms both are in place — `overlayUi` is validated and seed logic (`readSeedFromUrl`, `resolveSeed`, `writeSeedToUrl`) runs on load, ensuring the URL contains the `?seed=` parameter before the share button is used.

**Factory function pattern consistency:** The plan's `createShareButton(container: HTMLElement): HTMLButtonElement` signature exactly mirrors the established `createInfoButton(container: HTMLElement): HTMLButtonElement` pattern. The `createToast(container: HTMLElement): Toast` signature mirrors `createInfoModal(container: HTMLElement): InfoModal` — a factory returning an interface with methods. Both patterns are validated by the existing codebase.

**Toast as a reusable component:** Introducing `toast.ts` as a general-purpose notification system is well-scoped. The `Toast` interface (`{ show(message: string, type: 'success' | 'error'): void }`) is minimal and sufficient. Future tickets (e.g., refresh button feedback) can reuse this component. The CSS opacity transition for show/hide follows the same pattern used by `info-modal-backdrop` (opacity + visibility toggling), which is already proven in the codebase.

**Share action separation:** Extracting share logic into `src/ui/share-action.ts` as a pure async function is a sound design decision. It separates the browser API interaction (Web Share, Clipboard) from the button creation and event wiring in main.ts. This makes the share logic independently testable with mocked `navigator.share` and `navigator.clipboard`.

**Web Share API fallback chain:** The `navigator.share` → `navigator.clipboard.writeText` → error toast cascade is the correct approach. The plan's risk note about `navigator.share` existing but rejecting on desktop browsers is important — the implementation must catch both "API not available" (`typeof navigator.share !== 'function'`) and "API rejects" (user cancellation, `AbortError`) scenarios. The plan identifies this correctly.

**Clipboard API secure context requirement:** `navigator.clipboard.writeText` requires HTTPS or localhost. During local development with Vite (`localhost:5173`), this works. In production on Vercel, HTTPS is standard. The only failure scenario is plain HTTP deployment, which is outside the project's deployment target. The error toast provides adequate user feedback for this edge case.

**Button stacking and positioning:** The info button is positioned at `bottom: 24px; right: 24px` with a 44px height. The plan positions the share button at `bottom: 80px; right: 24px`, which provides 12px gap between buttons (80 - 24 - 44 = 12). This is adequate spacing for touch targets. On small mobile viewports (minimum 320px width), the right-aligned stack uses only 44px + 24px margin = 68px of horizontal space, which is well within bounds. Vertical stacking of two 44px buttons with gap occupies 112px from the bottom, leaving ample content area on even the smallest viewports (568px iPhone SE height).

**Toast z-index:** The plan correctly identifies that toast z-index must be above overlay children to remain visible when the modal is open. The modal backdrop uses `z-index: 100` within the overlay stacking context. The toast should use `z-index: 200` or similar to ensure visibility above all overlay siblings. This is a straightforward CSS decision with no architectural risk.

**Integration with main.ts:** The plan adds imports and initialization inside the existing `else` block where `createInfoButton` and `createInfoModal` are already wired. The addition is additive — create toast, create share button, add click listener. No restructuring of main.ts is needed. The current main.ts structure cleanly accommodates this extension.

**Test strategy:** Following the US-013 pattern of file-system-based tests (source existence, CSS property validation, import checks) plus jsdom-based component tests is consistent with the established testing philosophy. The plan includes mocking `navigator.share` and `navigator.clipboard`, which requires `vi.stubGlobal` in Vitest — a well-supported pattern. Testing the fallback chain (share → clipboard → error) covers the critical behavioral contract.

**Bundle impact:** Zero new dependencies. Five small files (two TS modules, one async function, two CSS files). Negligible bundle cost. The `jsdom` devDependency already exists in package.json for component testing.

**Key risks acknowledged:**
- Web Share API browser inconsistency is the primary runtime risk. The fallback chain is correctly designed to handle both "not available" and "rejected" cases. The plan should ensure that `AbortError` (user cancellation of share dialog) shows no toast or a neutral message rather than an error toast, since cancellation is intentional user action, not a failure.
- Clipboard API secure context requirement is well-identified with appropriate mitigation (error toast). No further fallback is practical or necessary.
- Button stacking arithmetic is correct for current button count (2). A third button (US-014 refresh) will need coordinated positioning — the plan's absolute positioning approach works but may benefit from a shared CSS variable for button spacing if the stack grows beyond 3. This is a future concern, not a blocker.
- Toast z-index layering above the modal is correctly identified. The implementation must set toast z-index higher than the modal backdrop's `z-index: 100`.

**No architectural concerns identified.** This is a self-contained UI feature that extends the floating controls system established by US-013 without modifying existing code beyond the expected integration point in main.ts. The toast component introduces a reusable feedback mechanism that future tickets can consume. No new dependencies, no build configuration changes, and no modifications to the domain layer.
