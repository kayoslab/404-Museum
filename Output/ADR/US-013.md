## ADR: US-013 – Create Floating Info Button

**Status:** Approved  
**Date:** 2026-04-23

### Decision

Approve the plan to implement a floating info button and info modal in `src/ui/`, styled via `src/styles/`, wired through `src/main.ts`, with file-system-based tests following established patterns.

### Architectural Assessment

**Alignment with project architecture:** The plan places UI components in `src/ui/` and styles in `src/styles/`, exactly matching the structure prescribed in CLAUDE.md. This will be the first code in `src/ui/`, establishing the directory that future floating controls (refresh button US-014, share button US-015) will follow.

**Overlay container compatibility:** The plan correctly identifies that `#overlay-ui` uses `pointer-events: none` (confirmed in `src/styles/shell.css`) and that every interactive child must set `pointer-events: auto`. The container is `position: fixed; inset: 0; z-index: 10`, so child elements inherit the fixed viewport positioning — the info button does not need its own `position: fixed`, though using it is harmless.

**Modal z-index strategy:** The modal must layer above the info button within the overlay. Since `#overlay-ui` is z-index 10 relative to the page, internal z-index values only compete with sibling overlay elements. This is straightforward and well-scoped.

**CSS transition on display:none:** The plan correctly flags that `display` is not animatable and recommends opacity + visibility or class-based toggling. This is the standard approach and avoids a common CSS animation pitfall.

**Touch target sizing:** The 44×44px minimum touch target noted in the plan aligns with WCAG 2.5.5 and Apple HIG guidelines. The circular button design accommodates this naturally.

**Test approach:** The plan uses file-system-based tests (checking file existence, CSS content, import statements) consistent with US-001/002/003 test patterns. The tests verify structural correctness without requiring a browser environment, which matches the established testing philosophy. The plan correctly notes that JS-injected elements won't affect index.html regex tests from US-003.

**Integration with main.ts:** The current `src/main.ts` validates `#overlay-ui` exists but does nothing with it. The plan adds imports and component initialization inside the existing validation block, which is the natural extension point. No structural changes to main.ts are needed.

**Bundle impact:** Zero new dependencies. Two small TypeScript modules and two CSS files. Negligible bundle cost.

**Key risks acknowledged:**
- The `pointer-events: none` inheritance is the primary integration risk — explicitly addressed in the plan with `pointer-events: auto` on interactive elements.
- US-003 test fragility around HTML regex patterns is mitigated by the fact that info button/modal elements are JS-injected, not added to index.html.
- Modal close mechanisms (×, backdrop, Escape) are standard patterns with no architectural risk.

**No architectural concerns identified.** This is a self-contained UI feature that builds on the established shell structure without modifying existing code beyond the expected integration point in main.ts.
